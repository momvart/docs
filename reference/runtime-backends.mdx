Leaf's analysis happens at runtime, not compile time. After you instrument your program with `leafc`, you pick a runtime backend that receives the instrumentation callbacks and decides what to do with them — solve constraints, record traces, or do nothing at all. Each backend is a shared library (`libleafrt*.so`) that you make discoverable via `LD_LIBRARY_PATH` before running your binary.

## How backends work

When your `leafc`-compiled binary starts, it dynamically loads whichever `libleafrt.so` is on `LD_LIBRARY_PATH`. The backend registers itself and then receives callbacks for every instrumented MIR operation — assignments, branches, function calls, and symbolic variable registrations. Swapping backends requires only rebuilding the desired backend and updating `LD_LIBRARY_PATH`; you do not need to recompile your program.

### Shared library naming

Each flavor produces a `cdylib` with a distinct file name. After building, the `.so` file appears as `target/debug/libleafrt_<name>.so` (or `target/release/` for release builds). Because the instrumented binary looks for a library named `libleafrt.so`, you create a symlink in a per-backend directory:

```bash
mkdir -p target/debug/<backend_name>
ln -sf "$PWD/target/debug/libleafrt_<name>.so" \
    target/debug/<backend_name>/libleafrt.so
export LD_LIBRARY_PATH="$PWD/target/debug/<backend_name>:$LD_LIBRARY_PATH"
```

Replace `<backend_name>` with the flavor package name (e.g., `runtime_symex`) and `<name>` with the library stem for that backend (e.g., `symex`).

## `runtime_symex` — Symbolic Execution

The symbolic execution backend performs full concolic analysis. It tracks symbolic values through your program, collects path constraints at every branch, and queries the Z3 SMT solver to find concrete input values that would take a different branch — the **diverging inputs**.

**Build command**

```bash
cargo build -p runtime_symex
```

**Set up and run**

```bash
mkdir -p target/debug/runtime_symex
ln -sf "$PWD/target/debug/libleafrt_symex.so" \
    target/debug/runtime_symex/libleafrt.so
export LD_LIBRARY_PATH="$PWD/target/debug/runtime_symex:$LD_LIBRARY_PATH"
export LEAF_LOG=info
./your_program
```

**Output**

- Binary files in `leaf_out/` — each file contains one set of concrete values for the symbolic variables, ordered by the sequence in which they were marked symbolic. These files can be piped directly to your program's stdin for re-execution.
- JSONL trace files in `leaf_out/` — control-flow paths, constraint formulas, and path preconditions.

**Features**

| Feature | Description |
|---|---|
| `implicit_flow` | Enabled in this flavor. Tracks implicit information flows through branches, giving the solver visibility into values that influence control flow indirectly. |
| `runtime_access_raw_ptr` | Enabled. Uses raw-pointer access for the variable state for performance. |
| `release_value_loss_checks` | Enabled. Adds assertions in release builds to detect value-loss bugs in the backend. |

<Note>
  This backend uses Z3, which can be computationally expensive. If the solver
  times out before finding a diverging input, see the
  [Troubleshooting guide](/reference/troubleshooting) for how to increase the
  timeout in `leaf_config.toml`.
</Note>

## `runtime_symex_no_implicit` — Symbolic Execution (No Implicit Flow)

This flavor is identical to `runtime_symex` except that implicit information flow tracking is **disabled**. Turning off implicit flow reduces the number of constraints the solver must handle, which can improve performance on programs where implicit flows are not the focus of the analysis.

**Build command**

```bash
cargo build -p runtime_symex_no_implicit
```

**Set up and run**

```bash
mkdir -p target/debug/runtime_symex_no_implicit
ln -sf "$PWD/target/debug/libleafrt_symex_no_implicit.so" \
    target/debug/runtime_symex_no_implicit/libleafrt.so
export LD_LIBRARY_PATH="$PWD/target/debug/runtime_symex_no_implicit:$LD_LIBRARY_PATH"
```

**When to use this over `runtime_symex`**

Use `runtime_symex_no_implicit` when you are analyzing programs for reachability or input coverage rather than information-flow security. The reduced constraint set typically leads to faster solver responses and fewer spurious diverging inputs.

## `runtime_cf_tracer` — Control-Flow Tracer

The control-flow tracer records execution paths without performing any constraint solving. It is a lightweight backend for understanding how your program actually executes — which branches are taken, in what order, and how often — without the overhead of symbolic reasoning or an SMT solver.

**Build command**

```bash
cargo build -p runtime_cf_tracer
```

**Set up and run**

```bash
mkdir -p target/debug/runtime_cf_tracer
ln -sf "$PWD/target/debug/libleafrt_cf_tracer.so" \
    target/debug/runtime_cf_tracer/libleafrt.so
export LD_LIBRARY_PATH="$PWD/target/debug/runtime_cf_tracer:$LD_LIBRARY_PATH"
```

**Output**

- JSONL trace files in `leaf_out/` — each line is a JSON object describing a branch or control-flow event observed during execution.

**When to use**

  <Card title="Path exploration" icon="route" href="/reference/output-formats">
    Understand which code paths a given input exercises before committing to
    full symbolic analysis.
  </Card>
  <Card title="Regression tracing" icon="code-branch" href="/reference/output-formats">
    Record execution traces for two runs and compare them to identify where
    behaviour diverged.
  </Card>
</CardGroup>

## `runtime_noop` — No-Operation

The no-op backend installs all the required runtime callbacks but leaves them completely empty. Your program runs to completion without any analysis overhead. No files are written to `leaf_out/`.

**Build command**

```bash
cargo build -p runtime_noop
```

**Set up and run**

```bash
mkdir -p target/debug/runtime_noop
ln -sf "$PWD/target/debug/libleafrt_noop.so" \
    target/debug/runtime_noop/libleafrt.so
export LD_LIBRARY_PATH="$PWD/target/debug/runtime_noop:$LD_LIBRARY_PATH"
```

**When to use**

Use `runtime_noop` to verify that `leafc` successfully compiles and instruments your program and that the binary loads and runs correctly, before adding the overhead of a real backend. It is also useful for measuring the instrumentation overhead itself by comparing execution times with and without the noop backend loaded.

## `runtime_mdsan` — ManuallyDrop Sanitizer

The `runtime_mdsan` backend implements a **ManuallyDrop sanitizer** — a dynamic analysis backend that tracks value lifetimes and memory operations to detect misuse of `std::mem::ManuallyDrop` and related unsafe memory patterns. It monitors place assignments and raw memory operations at the MIR level, flagging situations where values wrapped in `ManuallyDrop` are accessed after their logical lifetime has ended.

**Build command**

```bash
cargo build -p runtime_mdsan
```

**Set up and run**

```bash
mkdir -p target/debug/runtime_mdsan
ln -sf "$PWD/target/debug/libleafrt_mdsan.so" \
    target/debug/runtime_mdsan/libleafrt.so
export LD_LIBRARY_PATH="$PWD/target/debug/runtime_mdsan:$LD_LIBRARY_PATH"
```

**When to use**

Use `runtime_mdsan` when auditing unsafe Rust code that uses `ManuallyDrop`, raw pointer reads/writes, or custom allocators. It complements address sanitizers by focusing on logical lifetime violations rather than raw memory corruption.

## Switching backends

You can switch the active backend at any time without recompiling your instrumented binary. Build the new backend, create its symlink, and update `LD_LIBRARY_PATH`:

<Steps>
  <Step title="Build the new backend">
    ```bash
    cargo build -p runtime_cf_tracer
    ```
  </Step>
  <Step title="Create the symlink directory">
    ```bash
    mkdir -p target/debug/runtime_cf_tracer
    ln -sf "$PWD/target/debug/libleafrt_cf_tracer.so" \
        target/debug/runtime_cf_tracer/libleafrt.so
    ```
  </Step>
  <Step title="Update LD_LIBRARY_PATH and run">
    ```bash
    export LD_LIBRARY_PATH="$PWD/target/debug/runtime_cf_tracer:$LD_LIBRARY_PATH"
    ./your_program
    ```
  </Step>
</Steps>

## Backend comparison

| Backend | Constraint solving | Implicit flows | Output | Best for |
|---|---|---|---|---|
| `runtime_symex` | ✅ Z3 | ✅ Tracked | Binary + JSONL | Diverging input generation |
| `runtime_symex_no_implicit` | ✅ Z3 | ❌ Disabled | Binary + JSONL | Fast reachability analysis |
| `runtime_cf_tracer` | ❌ None | ❌ N/A | JSONL | Path tracing and comparison |
| `runtime_noop` | ❌ None | ❌ N/A | None | Compilation verification |
| `runtime_mdsan` | ❌ None | ❌ N/A | Log events | Unsafe memory auditing |
